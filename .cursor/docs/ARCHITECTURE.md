## Правила реализации бэкенда (Go, HTTP API, DDD)

Этот документ описывает единые правила разработки бэкенда проекта на **Go** с использованием стандартной библиотеки `net/http`, маршрутизатора **Gorilla mux** и **DDD**‑подхода. Все новые реализации должны строго соответствовать этим правилам.

***

## 1. Общая архитектура и слои

* **Версия Go**: как в `backend/go.mod` (модуль `github.com/scrumno/scrumno-api`).

* **Пакеты**: код организован по путям модуля; доменный код — под `internal/`, общие вещи — под `shared/`.

* **Фреймворк HTTP**: `net/http` + **Gorilla mux**; маршруты и middleware собираются в `internal/api/v1/routes.go` (функция `SetupRouter`).

* **Точка входа**: `backend/cmd/main.go` — загрузка конфигурации, подключение к БД, вызов `config.DI()`, запуск HTTP‑сервера.

* **Сборка зависимостей (DI)**: явная композиция в `backend/config/di.go` (функция `DI()`), возвращает агрегат `*action.Actions` с HTTP‑обработчиками и нужными сервисами.

* **Конфигурация окружения**: переменные окружения и при необходимости `.env` (см. `godotenv` в `main`); отдельные параметры — в пакете `config`.

* **DDD‑подход**:
  * Внешний мир (HTTP, внешние API и пр.) общается с доменом только через:
    * **Command**‑обработчики (`Handler`),
    * **Query**‑фетчеры (`Fetcher`),
    * при необходимости — отдельные **Service** внутри домена.
  * Нельзя вызывать инфраструктуру домена (GORM, сырой SQL, клиенты внешних API) напрямую из HTTP‑слоя — только через перечисленные фасады.
  * Агрегаты внутри домена опциональны; при прочих равных стараемся обходиться без сложных агрегатов.
  * Не усложняем модель value‑objects без острой необходимости.

***

## 2. Структура модулей и доменов

* **Корневая папка доменов**: `backend/internal/{domain}/` (например, `users`, `authorize`, `health`, `orders`).

* **Подпапки домена**:
  * `entity/{entity}/` — сущности и репозитории:
    * `{entity}.go` — модель сущности (часто с тегами GORM/`json`), при этом имя файла должно совпадать с именем папки сущности: `menu/menu.go`, `order/order.go`, `organization/organization.go`.
    * `{entity}-repository.go` (или `repository.go`) — доступ к данным.
    * Сущность и репозиторий всегда лежат в **разных файлах**.
    * **Исключение**: файлы `*-table.go` считаются легаси/инфраструктурным форматом и **не требуют** переименования под правило `{entity}.go`.
  * `command/{use-case}/` — команды:
    * `command.go` — DTO команды (структура).
    * `handler.go` — обработчик use‑case (`Handle`).
  * `query/{use-case}/` — запросы:
    * `query.go` (при необходимости) — DTO запроса.
    * `fetcher.go` — фетчер, возвращающий DTO или данные для ответа.
  * `service/` — доменные сервисы и вспомогательная логика.

* **HTTP‑слой**:
  * `internal/api/v1/http/action/{domain}/` — HTTP‑actions (обработчики конкретных эндпоинтов).
  * Структура `*Action` с методом, совместимым с `http.HandlerFunc` (или обёрткой вокруг него), плюс при необходимости типы входа для Postman‑коллекций.
  * Регистрация в `internal/api/v1/routes.go` и поля в `internal/api/v1/http/action/actions.go`.

***

## 3. HTTP‑слой (Actions)

* **Расположение**: `internal/api/v1/http/action/{domain}/`.

* **Роль Action**: принять `http.ResponseWriter` и `*http.Request`, разобрать путь/тело/заголовки, собрать Command/Query, вызвать `Handler`/`Fetcher`, вернуть JSON.

* **Конструктор**:
  * Зависимости передаются через конструктор `New...Action(...)` из `config/di.go`.
  * Поля обработчика — обычно указатели на `Handler`/`Fetcher` и т.п.

* **Поток обработки**:
  * Параметры маршрута — через `mux.Vars`, тело — через декодирование JSON или утилиты из `internal/api/utils`.
  * Создаём Command/Query (структуры с явными полями).
  * Вызываем `handler.Handle(ctx, cmd)` или `fetcher.Fetch(...)`.
  * Ответ — через `utils.JSONResponse` или аналогичный helper с нужным HTTP‑кодом.
  * Ошибки: по смыслу 4xx/5xx; валидационные и бизнес‑ошибки — предсказуемые сообщения для клиента.
  * Логирование — через стандартный `log/slog` (или принятый в проекте логгер); при ошибках — контекст и при необходимости трассировка.

* **Где уместны `defer` / обработка ошибок**:
  * В Action — нормализуем ошибки в HTTP‑ответы.
  * В репозиториях не «глотаем» ошибки без причины: возвращаем `error` наверх.

***

## 4. Команды (Command) и обработчики (Handler)

* **Расположение**: `internal/{domain}/command/{use-case}/`.

* **Command**:
  * Лёгкая структура без бизнес‑логики.
  * Поля экспортируются, типы явные.

* **Handler**:
  * Пакет по смыслу use‑case (часто `snake_case` в имени папки, например `create-user`).
  * Конструктор `New...Handler` принимает репозитории и сервисы.
  * Поля репозиториев именуем по смыслу (часто `repo` или конкретное имя зависимости).
  * Метод `Handle(ctx context.Context, cmd ...Command) (...)`:
    * Возвращаемый тип и `error` — явно.
    * Бизнес‑логика: сущности, репозитории, вызовы других доменных сервисов.
    * Прямых вызовов GORM/HTTP‑клиентов из Handler нет — только через репозитории и абстракции.

***

## 5. Запросы (Query) и фетчеры (Fetcher)

* **Расположение**: `internal/{domain}/query/{use-case}/`.

* **Query**:
  * По аналогии с Command — структура данных без логики.

* **Fetcher**:
  * Метод вроде `Fetch(query Query) Result` или с `context`, если нужно.
  * Возвращает DTO/структуру для ответа API.
  * Не знает о HTTP; только о домене и репозиториях/сервисах.

* **Регистрация в DI**:
  * `Fetcher` и `Handler` создаются в `config/di.go` и передаются в Actions явно.

***

## 6. Сущности (Entity)

* **Расположение**: `internal/{domain}/entity/{entity}/`.

* **Требования к сущности**:
  * Экспортируемая структура с тегами `gorm`/`json` по необходимости.
  * Идентификатор — обычно поле `ID` (в Go принят экспортируемый идентификатор; тип — например `uuid.UUID` или другой, принятый в проекте).
  * Конструкторы вида `New...` для создания новых сущностей — по месту, без лишней валидации в «чистом» конструкторе, если она относится к use‑case.
  * Маппинг из БД — через GORM; при ручном маппинге — явные преобразования типов без дублирования бизнес‑валидации в слое сущности, если она уже в Handler/Service.
  * Для сущностей интеграционных ответов используем принцип **минимально нужного состава полей**: состав полей определяется по согласованному списку, а не копированием полного payload провайдера.

***

## 7. Репозитории (Repository)

* **Расположение**: рядом с сущностью, например `entity/{entity}/{entity}-repository.go`, или обобщённая фабрика в `shared/factory` / интерфейсы в `shared/base`.

* **Два вида репозиториев**:
  * **Данные приложения (БД)** — чтение/запись в **PostgreSQL** через **GORM** (или согласованный слой доступа к данным).
  * **Интеграционные** — вызовы **внешнего HTTP API** (например iiko): запрос/ответ, маппинг в DTO/поля ответа; без GORM. См. подраздел ниже.

* **Общие правила**:
  * Маппинг «сырых» данных в доменные структуры — на границе репозитория.
  * Ошибки пробрасываются как `error`; при необходимости — оборачиваются с контекстом (`fmt.Errorf` с `%w`).

* **Сигнатуры методов** (аналогия с прежними соглашениями):
  * Если данные **обязаны** быть — имена вроде `Get...`; при отсутствии записи — договориться на уровне проекта (`ErrNotFound` или `error`).
  * Если допустимо отсутствие — `Find...` и указатель/`nil` или нулевое значение + флаг — по принятому в модуле стилю.
  * Коллекции: пустой слайс вместо `nil`, если так договорено для JSON/API.

* **Исключения / ошибки**:
  * Сообщения для пользователя API формируются в слое HTTP/Handler; в репозитории — технические и доменные ошибки через `error`.

### Интеграционные системы (отдельный домен + свой DI)

* **Домен интеграции** — отдельное дерево под `internal/{integration}/` (например `internal/iiko/`): команды, запросы, сущности, сервисы, **репозитории с HTTP‑запросами** к API поставщика.

* **Заменяемость поставщика (iiko/rkeeper и др.)**:
  * Сервис приложения должен зависеть от **интерфейса интеграции**, а не от конкретной реализации поставщика.
  * Для каждого поставщика (например `iiko`, `rkeeper`) реализуется свой домен интеграции с одинаковым контрактом интерфейса.
  * Выбор конкретной реализации делается в корневом DI через сборку нужного контейнера интеграции; бизнес‑код и HTTP‑actions не должны знать, какой именно поставщик подключён.

* **Конфиг интеграции** — в пакете `internal/{integration}/config/` (например `internal/iiko/config`): структура настроек и загрузка из окружения (`Load()`). Корневой `config.Load()` приложения **подставляет** этот конфиг в общую структуру приложения, чтобы точка входа в приложении знала параметры интеграции.

* **DI интеграции** — отдельная функция сборки внутри домена, например `internal/iiko/di.go` с `NewContainer(cfg *iikoconfig.Config) *Container`. Внутри неё собираются репозитории (HTTP), доменные сервисы и `Handler`/`Fetcher`, относящиеся к этой интеграции.

* **Точка входа в корневом DI** — в `backend/config/di.go` вызывается **одна** функция вида `iiko.NewContainer(&cfg.Iiko)`, а результат (например `SetAccess` и другие поля) **прокидывается** в `action.Actions` или в другие домены. Не размазываем создание репозиториев интеграции по корневому `di.go`: корневой файл только **вызывает** доменную сборку и передаёт результат.

* **Репозитории интеграции** — здесь живут **исходящие HTTP‑запросы** (`net/http` и т.п.), разбор JSON, коды ответов; обработчики команд и доменные сервисы вызывают репозитории, а не клиент напрямую.
  * Репозиторий возвращает **только сущности домена** (не raw JSON и не `map[string]any`).
  * Перед реализацией новой сущности по ответу внешнего API сначала фиксируем набор нужных полей: задаём вопрос «какие поля нужны из ответа?» и выбираем минимально достаточный набор.
  * В сущность включаем только выбранные поля; лишние поля внешнего ответа не маппим «на всякий случай».

***

## 8. Инфраструктура и границы

* **Доступ к БД и внешним API**:
  * К **БД** — только через репозитории и `shared/factory` (GORM).
  * К **внешнему API** — только через репозитории и HTTP‑клиенты **внутри** соответствующего домена интеграции (`internal/iiko/...` и т.д.) и конфигурацию подключений.
  * Доменные сущности и DTO **не** должны тащить за собой детали GORM/HTTP в публичный API пакета, кроме тегов на структурах, если это осознанный выбор слоя хранения.

* **Фоновые сценарии и события**:
  * Если появятся воркеры/очереди — граница такая же: вход в домен через команды/запросы/сервисы, без прямого SQL из «скриптов».

***

## 9. Конфигурация DI и окружения

* **Сборка зависимостей**: `backend/config/di.go` — создание репозиториев, handler’ов, fetcher’ов, передача их в `action.Actions`; для каждой **интеграционной системы** — один вызов её `NewContainer(...)`, без дублирования низкоуровневой сборки вне домена.

* **Конфиг приложения**: структура из пакета `config`, значения из переменных окружения (см. `config.Load()`). Поля интеграций типизируются конфиг‑пакетами доменов (`internal/{integration}/config`).

* **Что описываем в DI**:
  * Репозитории и обёртки над БД.
  * Доменные и инфраструктурные сервисы (JWT, SMS).
  * Сборку интеграций — через функции домена (`iiko.NewContainer` и аналоги).
  * Экземпляры `Handler`/`Fetcher` и Actions.

***

## 10. Стиль кода и соглашения

* **Типизация**: явные типы полей и возвращаемых значений; `context.Context` передаётся первым аргументом там, где есть I/O и отмена.

* **Экспорт**: что нужно снаружи пакета — с заглавной буквы; внутренние детали — с маленькой.

* **Имя пакета и алиасы импорта**:
  * Правило `first-second-third` → `first_second_third`: если папка называется в `kebab-case`, пакет в файлах этой папки называем в `snake_case` (нижний регистр, `_`), например `create-user` → `create_user`, `check-status-connect-db` → `check_status_connect_db`.
  * Если папка обычная (например `entity`, `config`, `service`) — не переименовываем пакет из-за этого правила: используем нормальное имя по смыслу.
  * При **импорте** пакета в `snake_case` задаём алиас в `camelCase`: `firstSecondThird` (например `create_user` → `createUser`, `check_status_connect_db` → `checkStatusConnectDb`).
  * Если в одном файле возникает коллизия двух одинаковых `camelCase` алиасов, базовый алиас сохраняем у приоритетного импорта, а для второго добавляем короткий доменный суффикс/префикс в `camelCase` (например `createUserAuth`).
  * Пример: `import createUser "…/internal/users/command/create-user"` — обращение к символам пакета: `createUser.NewHandler(...)`.

* **Ошибки**: идиоматичный Go — `if err != nil`; для бизнес‑случаев можно выделять сентинелы или типы ошибок в домене.
  * Тексты ошибок и комментарии к ошибкам (включая `fmt.Errorf(...)`, `errors.New(...)`, комментарии рядом с обработкой ошибок и ответы API с полем `error`) пишем **на русском языке**.

* **Логирование**: через `log/slog` (как в `main`) или единый логгер проекта.

* **Чистота кода**:
  * Неиспользуемые импорты удаляются (`goimports`/`gofmt`).
  * Зависимости только через конструкторы и параметры, без глобального состояния для бизнес‑логики.

***

## 11. Поток обработки HTTP‑запроса

* **Типичный сценарий**:
  1. Запрос попадает на `mux.Router` с префиксом `/api/v1/...`.
  2. Middleware (логирование, CORS, авторизация по JWT — см. `internal/api/v1/middleware`) обрабатывает запрос.
  3. Вызывается соответствующий Action.
  4. Action парсит вход, собирает Command/Query, вызывает `Handler`/`Fetcher`.
  5. Результат сериализуется в JSON; ошибки маппятся на коды ответа.

* **DDD‑границы**:
  * HTTP ↔ домены: через Command/Query и сервисы.
  * Домены ↔ БД/внешние системы: через репозитории и клиенты интеграций.

***

## 12. Дополнительные требования и запреты

* **Запрещено**:
  * Выполнять произвольные SQL/GORM‑запросы в HTTP‑handlers в обход репозиториев.
  * Смешивать разметку/шаблоны с доменной логикой в `internal/` (см. раздел 13).

* **Обязательно**:
  * Для новых фич повторять структуру существующих доменов: `entity`, при необходимости `command`/`query`/`service`, HTTP‑action и регистрация в `routes.go` и `di.go`.

***

## 13. Фронтенд и представления

* В рамках этого репозитория бэкенд — **API на Go**; разметка, SPA или статика живут **вне** слоя `internal/{domain}`.
* Доменные пакеты не должны содержать HTML/шаблоны как часть бизнес‑логики; клиент работает с API по контракту (JSON).

***

Соблюдение этих правил обязательно для всех новых реализаций в бэкенде. При доработке существующих модулей по возможности приводим код к этим соглашениям, не ломая текущую бизнес‑логику.
